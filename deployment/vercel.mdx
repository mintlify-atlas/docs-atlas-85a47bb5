## Introduction

Vercel is the recommended platform for deploying Toluca Altas Montañas. This guide walks you through the complete deployment process, from connecting your repository to configuring environment variables.

## Prerequisites

Before starting, ensure you have:

- A GitHub, GitLab, or Bitbucket account with your repository
- A Vercel account (sign up at [vercel.com](https://vercel.com))
- Supabase project credentials (URL, anon key, and service role key)
- Your database schema deployed to Supabase

## Deployment Steps

<Steps>
  <Step title="Import Your Project">
    1. Log in to [Vercel Dashboard](https://vercel.com/dashboard)
    2. Click **"Add New..."** → **"Project"**
    3. Select your Git provider (GitHub, GitLab, or Bitbucket)
    4. Authorize Vercel to access your repositories
    5. Find and import the `TolucaFC` repository
  </Step>
  
  <Step title="Configure Build Settings">
    Vercel will auto-detect Next.js. Verify the following settings:
    
    - **Framework Preset**: Next.js
    - **Root Directory**: `./` (leave as default)
    - **Build Command**: `npm run build`
    - **Output Directory**: `.next` (auto-detected)
    - **Install Command**: `npm install`
    
    <Note>
      These settings are usually pre-filled correctly. Only change them if you have a custom project structure.
    </Note>
  </Step>
  
  <Step title="Add Environment Variables">
    In the **Environment Variables** section, add the following:
    
    ```bash
    NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
    NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here
    SUPABASE_SERVICE_ROLE_KEY=your-service-role-key-here
    ```
    
    <Warning>
      Make sure to add `SUPABASE_SERVICE_ROLE_KEY` as it's required for the player registration API route.
    </Warning>
    
    See [Environment Setup](/deployment/environment-setup) for detailed information about each variable.
  </Step>
  
  <Step title="Deploy">
    Click **"Deploy"** to start the deployment process.
    
    Vercel will:
    1. Clone your repository
    2. Install dependencies
    3. Run the build command
    4. Deploy to the edge network
    
    The initial deployment typically takes 2-3 minutes.
  </Step>
  
  <Step title="Verify Deployment">
    Once deployed:
    
    1. Click the **"Visit"** button to open your live site
    2. Test the landing page registration form
    3. Test the admin panel login
    4. Verify database connections are working
    
    Your site is now live at: `https://your-project.vercel.app`
  </Step>
</Steps>

## Custom Domain Setup

To use a custom domain:

<Steps>
  <Step title="Access Domain Settings">
    1. Go to your project in Vercel Dashboard
    2. Navigate to **Settings** → **Domains**
  </Step>
  
  <Step title="Add Your Domain">
    1. Enter your domain name (e.g., `toluca-montanas.com`)
    2. Click **"Add"**
    3. Vercel will provide DNS configuration instructions
  </Step>
  
  <Step title="Configure DNS">
    Add the following records to your domain provider:
    
    **For root domain:**
    ```
    Type: A
    Name: @
    Value: 76.76.21.21
    ```
    
    **For www subdomain:**
    ```
    Type: CNAME
    Name: www
    Value: cname.vercel-dns.com
    ```
  </Step>
  
  <Step title="Wait for Verification">
    DNS propagation can take up to 48 hours, but usually completes within a few minutes.
    
    Vercel will automatically issue an SSL certificate once DNS is verified.
  </Step>
</Steps>

## Automatic Deployments

Vercel automatically deploys your application when you push to your repository.

### Production Deployments

Pushes to your **main** or **master** branch trigger production deployments:

```bash
git add .
git commit -m "Update feature"
git push origin main
```

Your changes will be live in 2-3 minutes.

### Preview Deployments

Pushes to other branches create **preview deployments** with unique URLs:

```bash
git checkout -b feature/new-feature
git push origin feature/new-feature
```

Each preview deployment:
- Has its own URL: `https://your-project-git-feature-new-feature.vercel.app`
- Can be shared with team members
- Uses the same environment variables as production
- Is automatically deleted when the branch is merged/deleted

## Environment Variables Management

### Update Environment Variables

To update environment variables after deployment:

<Steps>
  <Step title="Navigate to Settings">
    Project → **Settings** → **Environment Variables**
  </Step>
  
  <Step title="Edit Variables">
    Click the **"⋯"** menu next to any variable to edit or delete
  </Step>
  
  <Step title="Redeploy">
    After changing environment variables, redeploy your application:
    
    1. Go to **Deployments** tab
    2. Click **"⋯"** on the latest deployment
    3. Select **"Redeploy"**
    
    Or push a new commit to trigger automatic redeployment.
  </Step>
</Steps>

### Environment-Specific Variables

You can set different values for different environments:

- **Production**: Used for `main` branch deployments
- **Preview**: Used for all other branch deployments
- **Development**: Used for `vercel dev` local development

<Tip>
  Use Preview environments with a separate Supabase project for testing without affecting production data.
</Tip>

## Troubleshooting

### Build Failures

<AccordionGroup>
  <Accordion title="Module not found errors">
    **Problem**: Build fails with `Module not found` errors
    
    **Solution**:
    1. Ensure all dependencies are in `package.json`
    2. Try clearing Vercel cache: Settings → **"Clear Cache"**
    3. Redeploy the project
  </Accordion>
  
  <Accordion title="Environment variables not working">
    **Problem**: Application can't connect to Supabase
    
    **Solution**:
    1. Verify environment variables are set correctly
    2. Ensure variable names match exactly (case-sensitive)
    3. Check that `NEXT_PUBLIC_` prefix is used for client-side variables
    4. Redeploy after changing environment variables
  </Accordion>
  
  <Accordion title="TypeScript errors during build">
    **Problem**: Build fails with TypeScript errors
    
    **Solution**:
    1. Run `npm run build` locally to reproduce
    2. Fix TypeScript errors in your code
    3. Commit and push changes
    
    Or temporarily disable type checking in `next.config.ts`:
    ```typescript
    const nextConfig: NextConfig = {
      typescript: {
        ignoreBuildErrors: true,
      },
    };
    ```
  </Accordion>
  
  <Accordion title="Functions timing out">
    **Problem**: API routes return 504 timeout errors
    
    **Solution**:
    1. Check Supabase connection is working
    2. Verify database queries are optimized
    3. Consider upgrading Vercel plan for longer function timeouts
    
    Free tier has 10s function timeout, Pro has 60s.
  </Accordion>
</AccordionGroup>

### Common Issues

**Issue**: Images not loading
- **Cause**: Image optimization configuration
- **Fix**: Ensure `next.config.ts` image settings are correct

**Issue**: 500 error on API routes
- **Cause**: Missing `SUPABASE_SERVICE_ROLE_KEY`
- **Fix**: Add the service role key to environment variables

**Issue**: Authentication not working
- **Cause**: Supabase URL mismatch or CORS issues
- **Fix**: Verify `NEXT_PUBLIC_SUPABASE_URL` matches your Supabase project

## Performance Optimization

### Enable Analytics

Vercel Analytics provides insights into real user performance:

1. Go to **Analytics** tab
2. Enable **"Web Analytics"**
3. No code changes needed - automatically integrated

### Speed Insights

Monitor Core Web Vitals:

1. Enable **"Speed Insights"** in Analytics tab
2. View real-time performance metrics
3. Identify pages that need optimization

### Edge Functions

For optimal performance, consider converting API routes to Edge Functions:

```typescript
export const runtime = 'edge';

export async function POST(req: Request) {
  // Your API logic
}
```

Edge Functions run on Vercel's global edge network for lower latency.

## Rollback Deployments

If a deployment introduces issues:

<Steps>
  <Step title="Go to Deployments">
    Navigate to the **Deployments** tab
  </Step>
  
  <Step title="Find Working Deployment">
    Locate a previous deployment that was working correctly
  </Step>
  
  <Step title="Promote to Production">
    Click **"⋯"** → **"Promote to Production"**
    
    This instantly reverts your site to the selected deployment.
  </Step>
</Steps>
